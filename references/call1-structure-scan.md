# 调用 1：结构扫描

## 执行规则

本文是调用 1 的完整指令。读取后严格按本文操作，不要提前执行调用 2/3 的动作。

**输入**：合同全文
**输出**：路由卡 + 宏观层发现 + 中观层发现
**不输出**：条款级风险分析、推荐措辞、示范文本比对（调用 2/3 的事）

## 步骤

### 1. 合同类型识别

读取 `references/contract-routing.md`。按以下优先级判定：

1. 文件路径 → 子项目关键词命中（如 `sale/cases/` → 买卖合同）
2. 合同标题 → 12 类关键词匹配
3. 交易实质 → 标题与实质不一致时，以实质为准

混合型交易采用"主合同类型 + 配套协议类型"双标签。

### 2. Playbook 路由

读取 `config/playbooks/` 目录。合同类型是否命中增强映射表（contract-routing.md 第八章）？

- **命中** → 读取对应 playbook 的 `meta + context + tiers`（约 2k tokens）。记录匹配的版本（如 C 版：自身购买-非住宅）。
- **未命中** → 走通用路径。读取 `references/review-framework.md` 全文。

### 3. 宏观层扫描

以下四项逐条检查。输出发现，不输出整改方案：

| 检查项 | 判准 |
|:---|:---|
| 主体资格 | 甲乙双方是否具备签署能力？签约代表权限是否充分？是否存在已知的失信/被执行记录？ |
| 标的合法性 | 标的是否可合法交易？是否存在已知限制（如抵押、查封、租赁、预售证状态）？ |
| 程序完备性 | 是否需要审批/备案/登记/内部决议？合同是否约定了生效条件？生效条件是否可执行？ |
| 交易结构 | 付款→交付→担保→退出 是否形成闭环？是否存在结构断裂？ |

### 4. 中观层扫描

| 检查项 | 判准 |
|:---|:---|
| 合同形式 | 框架协议/正式合同/补充协议，是否匹配当前业务阶段？ |
| 格式条款 | 对免责/限责条款是否做了显著提示（加粗/单独签署）？提示说明义务是否可举证？ |
| 附件一致性 | 主合同与附件是否存在明确冲突？文件优先级是否约定？ |

### 5. 反馈纠错注入

读取 playbook 的 `feedbackCorrections.records[]`。筛选匹配当前合同类型 + 当前审查条款的历史纠正记录。

输出每条纠正的：日期 + 条款类型 + 原判定 + 用户纠正 + 纠正理由。

无匹配记录则输出"无历史纠正记录"。

## 输出格式

```
## 路由卡
- 合同类型：[12 类归属]
- 匹配 Playbook：[有/无] → [playbook 文件名 + 适用版本]
- 审查路径：[专项 / 通用]
- 审查立场：[甲方/乙方/中立]
- 审查口径：[克制/常规/强势]

## 宏观层扫描
- 主体资格：[发现]
- 标的合法性：[发现]
- 程序完备性：[发现]
- 交易结构：[发现]

## 中观层扫描
- 合同形式：[发现]
- 格式条款：[发现]
- 附件一致性：[发现]

## 历史纠正注入
- [如有匹配记录，逐条输出]
- [无记录则输出"无历史纠正记录"]
```

## 完成

调用 1 完成。将路由结果传递给调用 2。
**不要在本阶段输出条款级分析或推荐措辞。**
